# 画面構成・画面遷移・UI/UX方針

## MVP画面構成

### ⓪ ルートページ（Landing）`/`

未ログインユーザーが最初に訪れるランディングページ。アプリの魅力を短く伝え、「Yapeで現金を受け取れる」ことを強く訴求する。

| 項目 | 内容 |
| --- | --- |
| 役割 | ランディングページ（未ログインユーザー向け） |
| 目的 | サービスの価値を伝え、ログイン・登録へ誘導する |
| 訴求内容 | タスク完了で即時キャッシュバック、Yapeウォレットで受け取り、無料で始められる |
| 3ステップ | ① タスクを選ぶ → ② 条件を達成する → ③ Yapeで現金を受け取る |
| メインCTA | 「無料で始める」「Yapeで受け取る」 |
| アクション | 画面内のどこを押しても `/login` へ誘導する |

### ① ログイン・登録画面（Auth）`/login`

開発工数を抑えるため、認証画面は自作せず標準UIを利用する。

| 項目 | 内容 |
| --- | --- |
| 表示要素 | サービスロゴ（黄色ベース） |
| ログイン方法 | Googleログイン（NextAuth.js。メールログインは将来Email Provider追加で対応） |
| 遷移先 | ログイン成功後、Home画面へ |

### ② Home（Monlix iframe表示）`/home` ※メイン画面

ログイン後に最初に見る、アプリの中心となる画面。ヘッダーで所持ポイントと最低換金ポイントまでの進捗を見せ、画面下部にMonlixのiframeを大きく表示する。

| 項目 | 内容 |
| --- | --- |
| 目的 | 報酬状況を確認しながら、Monlix iframe内でタスクを実行してもらう |
| ヘッダー | 現在の所持ポイント、最低換金ポイントまでの進捗（例: `2,500 pts` / 最低換金 `10,000 pts`） |
| UI要素 | 黄色いプログレスバー、Monlix iframe表示エリア |
| タスク実行 | タスク詳細確認・外部サイト誘導・条件達成はMonlix iframe内で完結する |
| 報酬反映 | iframe内でタスク完了後、Webhook経由でポイントを更新する |

### ③ ウォレット・換金画面（Wallet）`/wallet`

ボトムナビの「財布アイコン」から遷移。貯めた報酬をYapeで受け取るための換金申請画面。

| 項目 | 内容 |
| --- | --- |
| 目的 | 所持ポイントの確認と換金申請 |
| 表示要素 | 現在の所持ポイントと換算レート（1,000 pts = S/ 1） |
| 入力項目 | Yape受取用電話番号（9桁・`9`始まり） |
| アクション | 「換金申請する」ボタン（最低換金ポイント 10,000 pts に達するまで押せない。`pending` 中の申請があるときも押せない） |
| 履歴 | 過去の換金申請ステータス一覧（処理中 Pendiente／送金完了 Pagado） |

### ④ 管理者ダッシュボード（Admin）

ユーザーには見せない運営者専用の管理機能。

| 項目 | 内容 |
| --- | --- |
| 対象／想定端末 | 運営者専用／PC |
| 目的 | ユーザーの換金申請を確認・処理する |
| 表示内容 | 申請者、Yape番号、申請金額、ステータス |
| 操作 | 送金完了後にステータスを更新 → Wallet側の表示が「送金完了」に変わる |

> ✅ **決定**: `/admin` は自作しない。管理者は **DBクライアント（TablePlus / pgAdmin等）** でHeroku Postgresに直接接続し、`withdrawals` テーブルを確認・更新する運用とする（詳細は [04-decisions.md](./04-decisions.md)）。

## 画面遷移フロー

```
⓪ ルートページ `/`
        ↓ 画面内をタップ
① ログイン・登録画面 `/login`
        ↓ ログイン成功
② Home `/home`
        ↓ iframe内でタスク実行
Monlix iframe内でタスク詳細確認・外部サイト誘導・条件達成
        ↓ Webhookでポイントが増える
② Home `/home`
        ↓ ボトムナビの財布アイコン
③ Wallet `/wallet`
        ↓ 換金申請
管理者がDBクライアント（TablePlus等）で確認
        ↓ Yape Empresaで手動送金し、ステータスを更新
③ Wallet `/wallet` の履歴が「送金完了」に更新
```

## MVPで作るべき画面一覧

| No. | 画面名 | ルート | 優先度 | 役割 |
| --- | --- | --- | --- | --- |
| ⓪ | Landing | `/` | 高 | 未ログインユーザー向けLP |
| ① | Auth | `/login` | 高 | ログイン・登録 |
| ② | Home | `/home` | 最高 | 報酬確認・Monlix iframe表示 |
| ③ | Wallet | `/wallet` | 高 | ポイント確認・換金申請 |

管理者向け機能（換金申請の確認・処理）は自作のAdmin画面ではなく、DBクライアントでHeroku Postgresに直接接続して代替する。

## MVPの基本方針

- 未ログイン時の入口は `/` に集約し、画面タップで `/login` へ誘導する
- 認証画面はNextAuth.js（Google Provider）を使い、自作しない
- ユーザー向け画面は白背景＋黄色アクセントで統一する
- タスク一覧・詳細確認・外部誘導はMonlix iframe内に任せる
- Home画面では所持ポイント・プログレスバー・Monlix iframe表示に集中する
- 最初は換金処理を完全自動化せず、管理者が手動でYape送金する
- 管理業務はデザインよりも運用効率を優先する（DBクライアントを使う）
- ユーザー体験の中心は「Monlixでタスク完了 → Webhookでポイント反映 → 換金申請」

## UI/UXワイヤーフレーム方針

MVPなのでFigmaで綺麗なデザインを作る必要はなく、紙とペン・Excalidrawなどの手書き風ツールで十分とされている。デザインはTailwind CSSなどのUIライブラリをそのまま当てて最速で組む方針。

最小構成としては以下の3画面でMVPが成立するとされていた（後に⓪〜③の4画面構成に整理）。

- 案件一覧（Home）
- 案件詳細（条件確認）
- マイページ（残高確認・Yape送金履歴）
